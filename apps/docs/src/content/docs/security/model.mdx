Terminal apps have full shell access to your machine. They can read your SSH keys, access your files, run any command as you. Warren was built from the start with security as a first-class constraint.

---

## Principles

### 1. Zero Trust by Default

Warren accepts no connections without explicit pairing. There is no "trust everyone on the LAN" mode. Every device that can open a shell must have been explicitly paired by the user.

### 2. Per-Pair Authentication

Every device pair (controller ↔ host) has its own independent trust relationship. Access to one host does not help an attacker gain access to another.

```
Phone ──(key A)──> Mac A
Phone ──(key B)──> Mac B
```

Key A and Key B are completely independent.

### 3. No Inherited Access

Access is never transitive. If Phone has access to Mac A, and Mac A has access to Mac B, Phone cannot reach Mac B through Mac A. Warren enforces this at the protocol level — the server only routes sessions to its own PTY manager, not to other hosts.

### 4. Privacy First

Warren does not log terminal sessions by default. Logging is opt-in. Warren does not collect telemetry. Warren does not send any data to external servers.

### 5. Local First

All connections default to the local network. Remote access requires explicit user configuration (Cloudflare Tunnel, Tailscale). Warren does not provide a relay server.

## Authentication (v0.1)

In v0.1, Warren uses a **static token**:
- Generated by the host when Warren starts (`crypto.randomUUID()`)
- Displayed as part of the QR code
- Client sends token in response to `auth:challenge`
- Constant-time comparison prevents timing attacks

This is functional but not cryptographically strong. The token is transmitted in plaintext over WebSocket (unless TLS is used via a tunnel).

<Aside type="caution">
v0.1 is a proof of concept. Do not expose your Warren host to the internet without a TLS-terminating tunnel (Cloudflare, Tailscale, etc.) in v0.1.
</Aside>

## Authentication (v0.2+)

v0.2 introduces **X25519 ECDH** key exchange:

1. During pairing, each device generates an X25519 key pair
2. Devices exchange public keys via the QR code / PIN flow
3. Both sides independently compute the shared secret: `ECDH(myPrivate, theirPublic)`
4. The shared secret is stored locally (SQLite in `~/.warren/devices.db`)
5. On reconnect, the client signs a server-provided nonce with its private key
6. The server verifies using the stored public key

The shared secret is never transmitted. The authentication proof is a signature, not the secret.

## Transport Security

In v0.1, Warren WebSocket connections are unencrypted unless you use a tunnel:

| Setup | Encryption |
|-------|------------|
| Local LAN (v0.1) | None |
| Cloudflare Tunnel | TLS (Cloudflare manages cert) |
| Tailscale | WireGuard (E2E encrypted) |
| v0.2 (planned) | AES-GCM with derived shared secret |

In v0.2+, terminal I/O will be encrypted end-to-end with AES-GCM using the ECDH-derived key, independent of transport security.

## Host-Side Controls

The Warren dashboard gives host operators full control:

| Control | Description |
|---------|-------------|
| Host mode toggle | Stop accepting all connections instantly |
| Device revocation | Remove a device's trust relationship immediately |
| Session visibility | See all active sessions with device attribution |
| Permission levels | Full shell or revoked (v1 scope) |

Revocation is immediate — no sync lag, no grace period.

## PWA Lock Screen

The phone PWA has an optional PIN lock. This is a **local** lock:

- Protects against someone picking up your phone
- Implemented with SHA-256 hash in localStorage
- Does **not** replace pairing authentication
- Does **not** disconnect active sessions

The lock screen prevents access to the PWA's UI. The underlying WebSocket connection may still be open.

## Threat Model

| Threat | Mitigated? | Notes |
|--------|------------|-------|
| Unauthorized connection on LAN | ✅ | Requires pairing + token |
| Token interception (LAN, v0.1) | ⚠️ | Use a tunnel with TLS |
| Token interception (v0.2+) | ✅ | Challenge-response, not token transmission |
| Transitive access | ✅ | Protocol-enforced, no routing between hosts |
| Physical phone access | ✅ (opt-in) | PIN lock screen |
| Host compromise | ❌ | Out of scope — Warren runs with user privileges |
| Supply chain | ✅ | Open source, every release is auditable |
